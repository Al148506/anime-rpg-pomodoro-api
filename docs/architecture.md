# Backend Architecture

## Scope

The backend is a Spring Boot 3 application for an Anime RPG Pomodoro Productivity App. It exposes a JWT-secured REST API where every productivity, progression, and waifu customization resource belongs to the authenticated user.

The implementation keeps the current base package: `com.alexdev.animerpgpomodoro`.

## Modular Structure

```text
src/main/java/com/alexdev/animerpgpomodoro/
  auth/
  user/
  task/
  category/
  pomodoro/
  session/
  progression/
  waifu/
  common/
  config/
```

## Layers

Each business module follows the same layered model:

```text
controller -> service -> repository -> entity
              mapper      dto
```

Controllers only handle HTTP concerns, request validation, and `ResponseEntity` creation. Services own business rules, authorization-by-ownership, transactions, and persistence orchestration. Repositories expose JPA persistence operations. DTOs are the only API payload types.

## Architectural Decisions

All IDs are UUID values generated by Hibernate with `GenerationType.UUID` and represented as strings to match the existing codebase.

All API resources use DTOs. JPA entities are never returned directly.

All user-owned queries are scoped by the authenticated user's email through repository methods such as `findByIdAndUserEmail`.

Timestamps are represented with `LocalDateTime` and maintained in services for consistency with existing entities.

Enums are used for finite domain vocabularies: `PomodoroType`, `PomodoroSessionStatus`, `AchievementType`, and `WaifuRarity`.

Critical write operations use `@Transactional` to keep session completion, XP updates, achievements, and unlocks consistent.

Flyway owns the database migration baseline through `src/main/resources/db/migration/V1__create_core_and_rpg_modules.sql`. Hibernate `ddl-auto=update` remains enabled to preserve the current local development behavior already present in the project.

## Backend Flow

1. The frontend authenticates through `/api/v1/auth/login` or `/api/v1/auth/register`.
2. The backend returns a JWT whose subject is the user email.
3. The frontend sends `Authorization: Bearer <token>` on protected requests.
4. `JwtAuthenticationFilter` sets the authenticated email in the Spring Security context.
5. Services resolve the current user and enforce ownership at query level.
6. Controllers return `ResponseEntity` with DTO responses and proper status codes.

## Conventions

Endpoint base path is `/api/v1`.

Create endpoints return `201 Created`.

Delete endpoints return `204 No Content`.

Read and update endpoints return `200 OK`.

Validation errors return `400 Bad Request` with field details.

Missing or non-owned resources return `404 Not Found` to avoid leaking resource existence.

Conflict errors return `409 Conflict`.
